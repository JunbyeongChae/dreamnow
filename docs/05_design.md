## 0. 문서 목적

이 문서는 두 목업 파일에 그려진 7개 화면(메인/기업소개/메뉴소개/메뉴상세/고객센터/로그인/회원가입)을 분석해, **여러 화면·여러 뷰포트에서 반복되는 요소를 공통 컴포넌트로 추출**하고 컬러·타이포·간격·반응형 규칙을 토큰화한 것이다. TDD 4-1의 `frontend/src/components/` 구조와 1:1로 대응시켜, 실제 구현 시 어떤 파일에 어떤 컴포넌트를 만들지 바로 참조할 수 있게 한다.

> v0.1에서는 데스크톱(1440px) 목업만 있어 PRD 5-1의 모바일/태블릿 요구사항을 "추정"으로 남겨뒀다. v0.2에서는 모바일(375px)·태블릿(768px) 반응형 목업을 반영해 8장을 실측 기반 스펙으로 교체했다.

## 1. 디자인 컨셉 / 톤앤매너

- **브랜드 성격**: 1976년부터 이어온 베이커리카페 — "오래됐지만 신뢰감 있는" 톤
- **색 기조**: 아이보리/크림 바탕 + 다크 브라운 텍스트 + 골드 포인트. 채도를 낮춘 웜톤으로 베이커리 특유의 따뜻함을 표현
- **타이포 기조**: 헤드라인은 굵고 임팩트 있는 국문 디스플레이 서체(Black Han Sans), 본문은 가독성 높은 Noto Sans KR
- **레이아웃 기조**: 여백을 넉넉히 쓰는 그리드 기반 섹션 반복 구조 (히어로 → 3분할 카드 → 리스트 → 푸터)

### 2-2. Tailwind 커스텀 토큰 제안 (TDD 2장: Tailwind CSS 채택)

TDD가 Tailwind 기본 브레이크포인트(`md`/`lg`)는 그대로 쓰기로 했지만, 위 컬러는 Tailwind 기본 팔레트에 없으므로 `tailwind.config.ts`에 별도 등록이 필요하다.

```ts
// tailwind.config.ts (제안)
theme: {
  extend: {
    colors: {
      primary: "#2b2620",
      accent: "#c8963e",
      "accent-soft": "#e3c581",
      "bg-warm": "#faf7f0",
      "text-muted": "#8a8070",
      "border-warm": "#eadfc6",
      "status-success": "#8fa88a",
      "status-pending-bg": "#f3e2b8",
      "status-pending-text": "#8a6a2f",
    },
  },
}
```

## 5. 공통 컴포넌트 분석

목업 7개 화면을 비교한 결과, 아래 요소들이 2개 화면 이상에서 **동일한 형태로 반복**되어 공통 컴포넌트화 대상이다. TDD 4-1 폴더 구조에 이미 자리가 있는 것은 경로를 그대로 표기했고, 구조에 없는데 반복 패턴이 확인된 것은 **[신규 제안]**으로 표시했다.

### 5-1. GNB 헤더 — `components/layout/Header.tsx`

- **등장**: 7개 화면 전체, 3개 브레이크포인트 전체
- **구성(데스크톱 1440px)**: 좌측 로고(BI.png, height 48px) / 중앙 네비(기업소개·메뉴소개·고객센터, 현재 페이지 `active` 시 하단 2px 골드 보더) / 우측 로그인·회원가입 버튼
- **변형(variant) 확인 — 반응형 목업 반영 후 3단계 폭 × 2종 상태로 확정**:

| 브레이크포인트 | 높이 | 로고 height | 중앙 영역 | 우측 영역 |
|---|---|---|---|---|
| 데스크톱(1440px) | 84px | 48px | 가로 네비(15px) | 비로그인: 로그인(아웃라인)+회원가입(골드) 버튼 / 로그인 후: 아이디+로그아웃 / 관리자 로그인 후: 아이디+로그아웃+관리자 모드 토글 아이콘 |
| 태블릿(768px) | 76px | 40px | 가로 네비(13px, 축소) | 데스크톱과 동일한 3가지 상태, 버튼 12px·패딩만 축소 |
| 모바일(375px) | 64px | 32px | **없음(숨김)** | 햄버거 아이콘(24×24, 3-line svg) 1개만 — 로그인 상태와 무관하게 항상 햄버거만(5-1 하단 참고) |

- **인증 페이지형(06 로그인·07 회원가입)**: 3개 브레이크포인트 모두 공통으로 우측 로그인/회원가입 버튼이 없다 — 데스크톱/태블릿은 네비만 노출, 모바일은 햄버거 아이콘만 노출(다른 페이지와 동일하게 유지되는 것으로 확인됨)
- **활성 탭 표시 방식 확정(OQ-D05 해소)**: 데스크톱 목업은 활성 링크에 `border-bottom:2px solid #c8963e`를 쓰고 반응형 목업의 태블릿 프레임은 텍스트 컬러만 바꾸는 식으로 서로 달랐으나, **텍스트 색상 변경(`color-accent`)만으로 전체 브레이크포인트를 통일**하기로 확정했다. 데스크톱 목업의 하단 보더 스타일은 채택하지 않는다
- **모바일 패널 확정(OQ-D01 해소)**: 햄버거 클릭 시 헤더 바로 아래로 펼쳐지는 **드롭다운(아코디언)** 방식이며, 오프캔버스·풀스크린이 아니다. 패널 내용은 **네비게이션 3항목(기업소개/메뉴소개/고객센터)만** — 로그인/회원가입 링크는 패널에 포함하지 않는다. 일반 사용자는 로그인이 필요한 액션(예: 상담글 작성)을 시도할 때 비로소 `/login`으로 유도되고, 로그인 화면에서 필요 시 `/signup`으로 이동한다(PRD 4-2 US-17 참고). 관리자는 데스크톱(태블릿 이상)에서만 로그인해 운영하는 것으로 가정하므로, 모바일 헤더/패널에는 로그인 진입 경로나 role 분기 UI가 전혀 필요 없다.
- **로그인 후·관리자 상태 UI 확정(OQ-D08 해소)**: 목업엔 로그인 전 상태만 그려져 있었으나 논의로 확정했다.
  - **일반 사용자 로그인 후**: 데스크톱/태블릿 헤더 우측의 로그인/회원가입 버튼 자리가 **아이디(사용자명) + 로그아웃 버튼**으로 전환된다.
  - **관리자 로그인 후**: 위 아이디+로그아웃에 더해 **관리자 모드 토글 아이콘**이 함께 노출된다. 각 화면의 등록/답변 버튼(배너·팝업 등록, 메뉴 등록, 공지사항 등록, 상담 답변)은 role이 admin이어도 **평소엔 숨겨져 있고, 이 토글을 켰을 때만** 인라인으로 나타난다(PRD 4-6 US-18). 실수로 관리 화면을 건드리는 것을 방지하고, 방문자 화면과 관리 화면을 시각적으로 분리하려는 목적이다.
  - 모바일은 관리자를 데스크톱 전용으로 가정하므로(OQ-D01) 이 상태들을 모바일 헤더/패널에 반영할 필요가 없다.

### 5-2. 푸터 — `components/layout/Footer.tsx`

- **등장**: 01~05 화면 (06 로그인, 07 회원가입에는 3개 브레이크포인트 전부 **의도적으로 없음** — 폼 중심의 짧은 페이지이므로 제외한 것으로 판단)

### 5-3. Button — **[신규 제안]** `components/common/Button.tsx`

목업 전체에서 버튼이 시각적으로 3가지 톤으로 일관되게 쓰인다. TDD 폴더 구조엔 별도 Button 컴포넌트가 없어 각 폼마다 스타일이 중복될 위험이 있다.

| Variant | 스타일 | 사용처 |
|---|---|---|
| `primary` (dark) | `bg-primary text-white` | 로그인/가입 제출, "문의 작성", 로그인 헤더 CTA(가입 시) |
| `accent` (gold) | `bg-accent text-white` | 헤더 "회원가입", 히어로 CTA는 예외적으로 흰 배경(아래 참고) |
| `outline` | `border border-primary` | 헤더 "로그인" |
| `text-link` (화살표) | `color-accent, font-weight 700, "→"` | "자세히 보기 →", "+ 공지 등록(관리자)" |

> 히어로 섹션의 "공지사항 보기" 버튼만 흰 배경+다크 텍스트로 예외 — 어두운 이미지 위 대비 확보 목적. `variant="inverse"`로 별도 추가 고려.

### 5-4. 이미지 슬롯(Placeholder) — **[신규 제안]** `components/common/ImageSlot.tsx`

목업의 `<image-slot>` 커스텀 엘리먼트가 5곳에서 반복 사용된다 (히어로, about 스토리, 메뉴카드×4, 메뉴상세). 실제 구현에서는 실제 업로드 이미지(`imageUrl`) + 로딩 스켈레톤(PRD 엣지케이스: "이미지 로딩 중 스켈레톤 UI") + 빈 상태를 모두 처리하는 공통 컴포넌트로 통합하는 것을 제안한다. TDD의 `common/Skeleton.tsx`와 함께 조합해서 사용.

| 사용처 | 비율/크기 |
|---|---|
| 메인 히어로 | 100% width × 360px height |
| about 스토리 | 1:1에 가까운 비율 × 420px height |
| 메뉴 카드(그리드) | 100% width × 160px height |
| 메뉴 상세 | 440×440px 정방형 |

### 5-5. Badge/Tag — **[신규 제안]** `components/common/Badge.tsx`

TDD 폴더 구조에는 없지만 목업에서 2가지 성격의 뱃지가 명확히 구분되어 반복된다. **하나의 컴포넌트로 통합하되 variant로 분기**할 것을 제안한다.

| Variant | 스타일 | 사용처 |
|---|---|---|
| `outline-accent` | 흰 배경, 골드 테두리+텍스트 | 공지 카테고리(신메뉴/휴무/채용) |
| `status-success` | `bg-status-success text-white` | 문의 "답변완료" |
| `status-pending` | `bg-status-pending-bg text-status-pending-text` | 문의 "답변대기" |

답변 상태는 TDD 6-4에 따라 `answered_at` null 여부로 프론트에서 파생 계산 후 이 컴포넌트에 전달한다.

- **반응형 확인**: 뱃지 자체 스타일은 3개 브레이크포인트에서 동일하나 크기만 축소된다 — 데스크톱 11px/패딩 3px 8px → 모바일·태블릿 10px/패딩 2px 6px. `status` 계열은 모바일에서 9px까지 더 축소.

### 5-6. 리스트 로우(List Row) — **[신규 제안, 패턴 공유]**

`support/NoticeList.tsx`와 `support/InquiryList.tsx`(TDD 기준 별개 컴포넌트)가 시각적으로는 **동일한 행 레이아웃**(뱃지/텍스트 + flex:1 제목 + 우측 정렬 날짜 or 상태)을 공유한다. 완전히 통합하기보다, 공통 `common/ListRow.tsx` 프레젠테이셔널 컴포넌트를 만들고 NoticeList/InquiryList는 이를 감싸 데이터만 주입하는 방식을 제안한다 — 두 리스트의 데이터 모델(Notice vs Inquiry)이 다르므로 완전 병합은 지양.

- **반응형 확인 — 날짜 필드가 브레이크포인트에 따라 노출/생략된다**:

| 브레이크포인트 | 공지사항 리스트 | 1:1 문의 리스트 |
|---|---|---|
| 데스크톱 | 뱃지+제목+날짜(`2026.07.15`) 모두 노출, 3열 카드 그리드 | 제목+상태뱃지+날짜, 900px 중앙 정렬 |
| 태블릿 | 뱃지+제목+날짜(`07.15`, 축약 포맷) 모두 노출, 560px 중앙 정렬 리스트형 | 제목+상태뱃지+날짜(축약), 560px 중앙 정렬 |
| 모바일 | **날짜 생략**, 뱃지+제목만 세로 스택 | **날짜 생략**, 제목+상태뱃지만, "문의 작성" 버튼 라벨도 "작성"으로 축약 |

`ListRow`는 `date` prop을 옵셔널로 받아 `null`/`undefined`일 때(또는 브레이크포인트가 모바일일 때) 날짜 영역을 렌더링하지 않는 형태로 설계한다.

### 5-10. 퀵링크 카드(Feature Card) — 메인 전용, 재사용 낮음

아이콘+제목+설명+"자세히 보기" 링크 카드는 메인페이지에만 등장한다. 목업에서는 내부의 **아이콘 세트**가 인라인 SVG(24×24 viewBox, stroke 방식)로 그려져 있지만, 실제 구현은 **`lucide-react`를 도입해 아이콘 라이브러리 기반으로 통일하기로 확정**했다(OQ-D02 해소, TDD 2장 참고). 헤더 햄버거·화살표 등 다른 UI 아이콘도 동일하게 `lucide-react`를 사용한다.

- **반응형 확인 — 모바일/태블릿은 콘텐츠 자체가 축약된다** (단순 리사이즈가 아니라 카드 안의 정보량이 브레이크포인트별로 달라짐):

| 브레이크포인트 | 그리드 | 카드 콘텐츠 |
|---|---|---|
| 데스크톱 | 3열 | 아이콘 + 제목 + 설명 문구 + "자세히 보기 →" 링크 |
| 태블릿 | 3열 | **아이콘 + 제목만** (설명·링크 제거) |
| 모바일 | 2열 (3번째 카드는 `grid-column:1/3`로 전체폭) | **아이콘 + 제목만** (설명·링크 제거) |

→ `QuickLinkCard`는 `variant="full" | "compact"` 정도로 분기하거나, 브레이크포인트 자체를 prop으로 받아 내부에서 설명/링크 렌더링 여부를 결정하는 방식을 제안한다.

- **히어로 CTA("공지사항 보기" 버튼)도 모바일/태블릿에서 생략** — 데스크톱에만 존재한다. 히어로 텍스트 오버레이 위치·그라디언트 방향도 브레이크포인트별로 다르다(8장 참고).

### 5-11. 카테고리 사이드바 — `components/menu/CategorySidebar.tsx`

메뉴소개 전용, TDD에 이미 정의됨. 상위 카테고리(시즌메뉴/음료/디저트) 중 음료만 하위 카테고리(coffee/non-coffee/tea/ade·juice)를 들여쓰기 리스트로 펼치는 2단 구조를 데스크톱 목업에서 확인. PRD 4-4 요구사항과 정확히 일치.

- **반응형 확인 — PRD 4-4 "모바일은 상단 가로 스크롤 탭(또는 드롭다운) 전환"이 목업으로 구체화됨**:

| 브레이크포인트 | 형태 |
|---|---|
| 데스크톱(260px) / 태블릿(180px) | 좌측 고정 사이드바, 음료 하위 카테고리는 들여쓰기 리스트로 항상 펼침 |
| 모바일 | `overflow-x:auto` 가로 스크롤 탭 바(세로 사이드바 완전히 대체). 활성 탭(`시즌메뉴`)은 다크 배경(`#2b2620`)+흰 텍스트 필박스, 비활성 탭(`음료 ▾`, `디저트`)은 `1px solid #ddd` 아웃라인 + 회색 텍스트. **"음료 ▾" 탭 클릭 시 확정(OQ-D06 해소)**: 탭 바 바로 아래에 두 번째 줄로 coffee/non-coffee/tea/ade·주스 하위 탭 4개가 아코디언 형태로 펼쳐짐(새 레이아웃·모달 없이 기존 탭 바와 동일한 스타일 유지) |

TDD 6-5 각주가 요구한 대로 "탭 ↔ 고정 사이드바"는 CSS `display` 토글이 아니라 `useBreakpoint` 훅으로 컴포넌트 자체를 분기 렌더링해야 한다.

- CTA/확인 버튼이 있다면 5-3 Button의 `accent` variant를 재사용

### 5-14. 배너 슬라이더 — `components/banner/BannerSlider.tsx` (OQ-D04 해소)

- 배너가 1장뿐일 때는 화살표·인디케이터를 숨기고 정적 이미지처럼 노출(4-1 빈 상태와 별개로, "배너 1장"은 정상 케이스)

### 8-2. 컴포넌트 단위 반응형 원칙

| 컴포넌트 | 처리 방식 |
|---|---|
| Header | `useBreakpoint` 훅으로 분기 렌더링 — 모바일은 기본적으로 로고+햄버거만 렌더링, 클릭 시 헤더 아래로 네비 3항목(로그인/회원가입 링크 없음)이 드롭다운(아코디언)으로 펼쳐짐. 태블릿↑은 로고+네비+로그인/회원가입 버튼 풀렌더링. 관리자는 데스크톱 전용으로 가정해 모바일 분기에 role 처리 불필요(OQ-D01 해소, 5-1 참고) |
| CategorySidebar | 모바일은 `CategoryTabBar`(가로 스크롤), 태블릿↑은 `CategorySidebar`(좌측 고정) — 서로 다른 컴포넌트로 분기, CSS `display:none` 토글 금지(TDD 6-5) |
| QuickLinkCard | `variant` 또는 브레이크포인트 prop으로 설명·링크 렌더링 여부 분기 (5-10) |
| ListRow(Notice/Inquiry) | `date` prop이 없거나 모바일일 때 날짜 영역 미렌더링 (5-6) |
| Footer | 모바일은 전화번호 제외 축약 콘텐츠 (5-2) |
| MenuDetail 레이아웃 | 모바일·태블릿은 `flex-direction:column`, 데스크톱은 `flex-direction:row` — 컴포넌트 교체 없이 CSS만으로 처리 가능 |
| 히어로 CTA / 퀵링크 설명·링크 | 모바일·태블릿에서 완전히 숨김(단순 축소가 아니라 미노출) — 조건부 렌더링 필요, `display:none`만으로 처리 시 스크린리더에 불필요한 DOM이 남으므로 조건부 렌더링 권장 |
